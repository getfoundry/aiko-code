## openclaw/openclaw#gateway-startup-architecture [2026-04-30T04:00:00Z]
Main entry point is `startGatewayServer()` in `src/gateway/server.impl.ts`. Startup sequence: (1) Load config snapshot via `loadGatewayStartupConfigSnapshot` -> `prepareGatewayStartupConfig`, (2) Bootstrap auth, (3) Bootstrap plugins via `bootstrapGatewayPlugins()` which creates empty registry then calls `loadGatewayStartupPlugins()` returning a `pluginRegistry` with `gatewayHandlers` and `baseGatewayMethods`, (4) Create runtime state with `createGatewayRuntimeState()` which returns WS server (wss) and client tracking structures, (5) `startGatewayEarlyRuntime()` — initializes discovery service (Bonjour/mDNS), skills registry (remote node skills caching + task registry), skills change listener (debounced refresh of remote bins), maintenance timers (health snapshots, presence updates, chat buffering, deduplication), (6) `attachGatewayWsHandlers()` — merges plugin `gatewayHandlers` with core handlers, registers all gateway methods, sets up handshake protocol with auth challenges, configures rate limiting, (7) Listen on port, (8) `startGatewayPostAttachRuntime()` — loads internal hooks, starts channels via `startChannels()`, starts plugin services. Channels load in two phases: Phase 1 `loadGatewayStartupPlugins()` loads channel plugins configured to start immediately; Phase 2 `startChannels()` (called post-attach) activates remaining configured channels — deferred so WS server accepts connections first. `createChannelManager()` in `src/gateway/server-channels.ts` handles channel lifecycle. Note: "QueryEngine session lanes" term not found in visible codebase; `applyGatewayLaneConcurrency()` exists but QueryEngine implementation not shown.


## openclaw/openclaw#ws-protocol-frames [2026-04-30T04:00:00Z]
No single `connection-handler.ts` — implementation split across `ws-connection.ts` (initial WS connection setup) and `message-handler.ts` (all frame parsing/routing). Four frame types, all JSON-encoded: (1) Connect (handshake): `{type:"req", method:"connect", params: ConnectParams}`, (2) Request: `{type:"req", id, method, params}`, (3) Response: `{type:"res", id, ok, payload|error}`, (4) Event: `{type:"event", event, payload, seq?, stateVersion?}` (server push). Handshake flow: server sends `connect.challenge` event with nonce -> client responds with `connect` request -> server validates and sends `hello-ok` response with policy limits -> bidirectional req/res and event frames follow. Inbound validation: parses JSON, extracts frame type/method/id for logging. Preauth payload limit: 64 KiB cap on frames before handshake. Post-handshake: clients receive `hello-ok.policy` with `maxPayload` and `maxBufferedBytes` limits; oversized frames emit `payload.large` diagnostic events then connection closes. No explicit chunking at WS frame level — large responses sent as single JSON frames up to `maxPayload`. Streaming uses multiple event frames with `seq` numbering for ordering; client detects gaps and invokes `onGap` callback for out-of-order frames. Request/response correlated by `id` field — client maintains `pending` map keyed by request ID, resolving promises on matching response. For streaming methods with `expectFinal` flag, intermediate ack frames with `status:"accepted"` held until final response. Handshake timeout configurable via env var.

## openclaw/openclaw#service-lifecycle [2026-04-30T04:00:00Z]
No single `service-manager.ts` — service lifecycle distributed across `src/cli/daemon-cli/` modules: `lifecycle.ts` registers CLI commands (status, install, uninstall, start, stop, restart), `lifecycle-core.ts` contains `runServiceStart`/`runServiceStop`/`runServiceRestart`, and `restart-health.ts` handles post-restart verification. `runServiceStart` validates service is loaded, runs preflight config checks, calls `startGatewayService`. `runServiceStop` sends stop signals. `runServiceRestart` orchestrates restart with post-restart health verification. Health checks via `inspectGatewayRestart` check: process runtime status (running/stopped), port usage (busy/free), RPC reachability and gateway version, stale gateway PIDs needing termination. Result includes `healthy`, `staleGatewayPids`, `portUsage`, `runtime`. Unmanaged gateway handling: for unmanaged restarts, sends `SIGUSR1` directly to gateway process PID. Post-restart verification: system waits for gateway to become healthy by polling port and checking RPC connectivity; if stale PIDs detected, terminates them and retries restart. Gateway daemon integration: `startGatewayServer()` in `src/gateway/server.impl.ts` initializes core gateway server during startup (loads config, sets up auth, initializes WS handler, starts channel management). `channelManager` provides `startChannels`, `startChannel`, `stopChannel`, `markChannelLoggedOut` — used in close handler and runtime services. When gateway restarts/stops, channels are properly shut down and restarted. Separation: CLI commands manage the daemon process lifecycle, gateway server manages internal subsystems including channels.

## anthropics/anthropic-sdk-typescript#streaming [2026-04-30T02:50:00Z]
Streaming uses `MessageStream` class with SSE events: `message_start`, `content_block_start`, `content_block_delta`, `message_delta`, `content_block_stop`, `message_stop`. The SDK parses SSE via `LineDecoder` + `SSEDecoder`. Retries are HTTP-level only (408/409/429/5xx) — no mid-stream retry. "Context limit reached" would be an HTTP 400 with no special SDK handling. The SDK does NOT have a concept of `assistant_turn_chunk` — that's aiko-code's own streaming abstraction in its API layer, not upstream.
